## 2. 실패 유형 분류 통합

### 2.1 유형별 발생 빈도

| 유형 | 코드 | set-C | set-D | set-E | set-F | 합계 |
|------|------|-------|-------|-------|-------|------|

| 소급 정책 미적용 | R | 5 | 2 | 6 | 4 | **17** |

### 2.2 핵심 발견

1. **API 시그니처 불일치(A)가 가장 빈번** (21건)
   - 프롬프트에 명시되지 않은 파라미터(예: `id=` 키워드), 반환값, 에러 처리 정책 차이
   - **문제점**: 이는 "스파게티 유도"가 아니라 "명세 불충분"에 의한 실패
   - **조치 필요**: 프롬프트에서 생성자 시그니처, 에러 처리 정책을 더 명확히 기술

2. **소급 정책 미적용(R)이 안정적으로 작동** (17건)
   - 대소문자 무시, 검증 규칙 소급 등이 기존 코드에 미적용
   - **이것이 진정한 스파게티 유도 패턴** — AI가 새 코드에만 적용하는 경향

3. **코드 경로 누락(P)이 예상대로 발생** (18건)
   - 예약/비예약 이중 경로, 완료 추적의 update vs move 경로 등
   - **유효한 스파게티 지표**

### 3.2 수정 필요 사항 (HIGH 우선순위)

#### Issue 1: set-C step5 — Cart 예약 매핑 명시
- **현재**: 암묵적으로 `{product_id: reservation_id}` 관리 필요
- **수정**: "ShoppingCart는 내부적으로 `_reservations: dict[str, str]` (product_id → reservation_id) 매핑을 관리합니다" 추가

#### Issue 2: set-D step3 — resource=None 동작
- **현재**: resource가 None일 때의 권한 체크 동작 미기술
- **수정**: "resource가 None이면 generic 권한만 검사합니다 (소유권 체크 생략)" 추가

#### Issue 3: set-E step7 — 수정 대상 프로세서 명시
- **현재**: "모든 프로세서를 수정하세요"만 기술
- **수정**: "다음 클래스를 수정하세요: TransformProcessor, EnrichProcessor" 명시

#### Issue 4: set-F step4 — 스레드 루트 탐색
- **현재**: "parent의 parent_id가 있으면 root로 매핑" 설명만
- **수정**: 의사코드 추가
  ```
  # 루트 찾기
  root_id = parent_id
  while get_message(root_id).parent_id is not None:
      root_id = get_message(root_id).parent_id
  ```

### 3.3 "스파게티 유도" vs "명세 불충분" 구분 필요

현재 실패 중 상당수는 **스파게티 코드가 아니라 프롬프트의 명세 부족**으로 발생:

| 실패 유형 | 스파게티 유도? | 예시 |
|-----------|:------------:|------|
| 방어적 복사 누락 | **O** | tags/metadata 프로퍼티에서 참조 반환 |
| 대소문자 무시 소급 미적용 | **O** | 기존 SchemaRegistry에 미적용 |
| 예약→비예약 이중 경로 | **O** | cart 없이 생성된 주문에서 재고 미차감 |
| 모델 생성자 id= 미지원 | **X** | 프롬프트에 id 파라미터 미명시 |
| Enum 값 대소문자 | **X** | "ONLINE" vs "online" 미명시 |
| 에러 처리 정책 (None vs 예외) | **X** | not found 시 동작 미명시 |
| AuditLog 키 네이밍 | **X** | "phase" vs "stage" 미명시 |

**권장**: 명세 불충분에 의한 실패를 최소화하고, 진정한 스파게티 유도 실패만 남기도록 프롬프트를 개선해야 함.

| 암호화 저장 (S) | 8 | Critical | 40% — in-place 복호화로 저장 오염 |
