# AI 코딩 에이전트 구조적 코드 품질 평가 시험

## 1. 시험 개요

### 목적

AI 코딩 에이전트가 **순차적으로 변화하는 요구사항** 하에서 **구조적으로 건전한 코드**를 유지할 수 있는지 평가한다.

### 핵심 가설

> AI 코딩 에이전트는 단일 프롬프트 내에서는 깔끔한 코드를 생성하지만,
> 후속 단계에서 기존 코드의 **구조적 변경이 필요**한 경우 스파게티 코드를 유발한다.

### 시험 형식

| 항목 | 내용 |
|------|------|
| 형식 | 순차적 프롬프트 기반 코드 생성 (8단계) |
| 언어 | Python |
| 평가 도구 | pytest (자동 채점) |
| 시험 셋 | 4개 (C, D, E, F) — 각각 독립된 도메인 |

## 3. 시험 진행 방법 (응시자 가이드)

### 3.1 환경 준비

```bash
git clone <repository-url>
cd ai-sw-certificate
python3 --version   # 3.9+ 필요
pip install pytest
```

### 3.2 시험 실행 절차

```bash
# 1) 셋 선택 후 이동
cd set-D

# 2) src/ 초기화 (레퍼런스 구현 제거)
cd src && for f in *.py; do [ "$f" != "__init__.py" ] && echo "" > "$f"; done && cd ..

# 3) AI에게 프롬프트 순차 제공
#    prompts/step1.md → step8.md 순서대로 하나씩 전달
#    AI가 src/ 에 코드를 생성/수정하도록 함

# 4) 최종 평가
python3 -m pytest tests/ -v                    # 전체 테스트
python3 -m pytest tests/test_final.py -v       # 통합 테스트 (핵심)
```

### 3.3 중요 규칙

- 프롬프트는 **반드시 순서대로** 하나씩 제공
- AI에게 **테스트 파일을 보여주지 않음**
- AI에게 추가 힌트나 수정 지시를 하지 않음
- 동일한 AI 세션 내에서 모든 단계를 진행 (컨텍스트 유지)

## 4. 채점 방법 (평가자 가이드)

### 4.1 자동 채점 스크립트

```bash
#!/bin/bash
# run_exam.sh <set-name>
SET=${1:-"set-D"}
cd "$SET"

echo "=== 전체 테스트 ==="
python3 -m pytest tests/ -v --tb=short 2>&1 | tee /tmp/exam_full.txt | tail -1

echo "=== 통합 테스트 (핵심) ==="
python3 -m pytest tests/test_final.py -v --tb=short 2>&1 | tee /tmp/exam_final.txt | tail -1

echo "=== 단계별 ==="
for i in $(seq 1 8); do
  result=$(python3 -m pytest tests/test_step${i}.py -q --tb=no 2>&1 | tail -1)
  echo "  Step $i: $result"
done
```

### 4.2 핵심 지표

| 지표 | 의미 |
|------|------|
| **전체 통과율** | 기본 기능 구현 능력 (step1~8 + final) |
| **test_final 통과율** | **구조적 품질 핵심 지표** — cross-cutting 통합 검증 |
| **후반부 통과율** (step5~8) | 리팩토링/구조 변경 대응 능력 |

### 4.3 등급 기준 (test_final 기준)

| 등급 | test_final 통과율 | 해석 |
|------|:--:|------|
| **A** | 90%+ | 우수 — 구조적 리팩토링 능력 입증 |
| **B** | 75~89% | 양호 — 대부분의 통합 시나리오 처리 |
| **C** | 60~74% | 보통 — 일부 cross-cutting concern 누락 |
| **D** | 40~59% | 미흡 — 주요 구조 변경 실패 |
| **F** | 40% 미만 | 불합격 — 심각한 스파게티 코드 발생 |

### 4.4 실패 유형 분류

| 코드 | 유형 | 예시 |
|:--:|------|------|
| R | 소급 누락 | 대소문자 무시가 새 코드에만 적용 |
| P | 경로 누락 | update로 상태 변경 시 완료 추적 안 됨 |
| D | 이중 실행 | Command가 public 메서드를 호출하여 훅 이중 실행 |
| S | 동기화 누락 | 메시지 수정 시 검색 인덱스 미갱신 |
| B | 하위 호환 | 새 직렬화 형식만 지원 |
| C | 방어적 복사 | list/dict 참조를 그대로 반환 |
| A | API 불일치 | 메서드 시그니처/반환값 차이 |

## 5. 셋별 상세 설계

| Step | 주제 | 함정 |

| 2 | Processor + Pipeline | — |

## 8. FAQ

**Q: AI에게 테스트 파일을 보여줘도 되나요?**
A: 아니요. 테스트를 보여주면 구조적 품질이 아닌 테스트 맞춤 능력을 측정하게 됩니다.

**Q: 프롬프트를 한 번에 모두 주면 안 되나요?**
A: 안 됩니다. 순차적 제공이 핵심입니다.

**Q: AI가 이전 단계의 코드를 수정해도 되나요?**
A: 네. 이것이 시험의 핵심입니다.

**Q: 어떤 셋을 먼저 시험해야 하나요?**
A: 어떤 셋이든 무관합니다. 도메인 친숙도에 따라 선택하세요.

**Q: 반복 시험 시 결과가 달라지나요?**
A: 네. 3회 이상 반복하여 평균을 권장합니다.
